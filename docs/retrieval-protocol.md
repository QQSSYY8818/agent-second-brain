# 检索协议（≤8 次读取封顶）

检索的第一原则：**不是"找得到"，而是"用最少的 token 找到"**。全库扫描永远找得到，但那是最贵的路。本协议给每一层设硬预算。

## 0. 递降检索主干（关键词入脑）

| 跳 | 动作 | 预算 |
|----|------|------|
| 第 0 跳 | 读/grep `_索引/_路由地图.md`（每项目：定位+关键词+承重卡 top-5，按 度+3×used 排序）→ 定关联最强项目 | ≤40 行 |
| 第 1 跳 | 读该项目 枢纽现况节 + MOC 结论摘要节（禁全文） | 各 ≤10 行 |
| 第 2 跳起 | 顺边读卡：先（强）边和高 used 边，（弱）边只在强边失准时才走 | 每跳 ≤2 张，可至 4-5 跳 |
| 全程 | **累计 ≤8 次文件读取铁顶** | 触顶=路由失败，记 `_检索失败日志.md`，回退 grep 总目录兜底 |

## 1. V 字跳（跨项目侧向发现）

任一跳落在实例卡上时，可走 `上位::` 边升到家族概念卡 → 邻接表反查该家族的其他下位实例 → 降入**别的项目**里关联更强的卡。升+降各计 1 次读取，仍受 8 次总额约束。**当前项目内查不顺时优先想这一手，而不是加深本项目内的跳数**——这是机械化的"联想"。

## 2. 题型路由表（按问题长相选入口）

| 问题长相 | 首选入口 |
|---|---|
| X 的最新状态/进展 | 枢纽-X 现况节 |
| 为什么这样设计/走过什么弯路/什么方案被否 | MOC-X 决策线节 + 事例卡 |
| 要具体数字/公式/结论 | grep 总目录 → 概念卡 |
| 做过类似的事吗 | 编年层 → 事例卡 |
| 还有什么没解决 | 问题卡（q_status=open）+ 枢纽开放问题节 |
| 这个词是什么意思 | 知识点卡（环外知识平面） |

## 3. 四层检索半径（从便宜到贵）

1. **L0 注入层**（零成本）：`brain_inject.py` 已把 top-4 相关卡描述放进上下文——先看注入块。
2. **L1 grep 总目录**：`_索引/_卡片总目录.jsonl`（name/aliases/description 一击定位）。命中即直读卡文件。
3. **L2 邻接表**：反向遍历与多跳查询一律读 `_索引/_邻接表.jsonl`，**不逐卡 grep**。
4. **L3 语义兜底**：目录与邻接表都 miss 时 `python engine/vault_semantic_search.py "问题"`（需 Ollama+bge-m3）。
5. **L4 外部检索**（最贵，miss 升级）：网搜/文献库核实（带 URL）→ **必须回写成卡**（confidence=文献或联想，外部事实带「截至 YYYY-MM」）→ **新卡必须接进降链**：与沿途摸到的最近相关卡挂边，理由写明"为什么当时没答上"。外找不建卡=白找；建卡不接链=孤岛复发。

## 4. miss 纪律

- 找不到答案 → `_检索失败日志.md` 追加一行（日期/问题/缺什么）。miss 日志是月度巡检的原料——反复 miss 的主题就是该建卡的主题。
- 命中已取代卡 → 必须配对返回其 `取代者::` 目标，引用旧结论须声明"已被 X 推翻"。

## 5. 用后记账（Hebbian 闭环）

一场任务真实调用过的卡，收口时跑 `python engine/vault_touch.py 卡名1 卡名2 ...`（或由收割钩子自动完成）——used+1 让这张卡下次浮得更高。**漏记=权重回路空洞**。
